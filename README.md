# Football League DB (PostgreSQL)

Проектная работа по дисциплине «Базы данных».  
База данных моделирует любительскую футбольную лигу с командами, игроками, руководителями и результатами матчей.

## Цель и предметная область

Цель проекта — спроектировать и реализовать реляционную базу данных для управления любительской футбольной лигой: хранение информации о командах, составах, матчах, результатах и статистике.
Модель поддерживает две лиги (первая и вторая), продвижение и вылет команд, учёт голов, побед, ничьих и поражений.

## Структура данных

Основные таблицы:

- `lfl_teams` — все команды лиги, их названия и уровень соревнований (первая/вторая лига).
- `first_league`, `second_league` — турнирные таблицы лиг с играми, победами, ничьими, поражениями и очками.
- `team_owners` — руководители команд и контакты (телефон, e‑mail).
- `lfl_players` — игроки, их ФИО и принадлежность к команде.
- `match_results` — результаты матчей между командами с датой встречи.

Примеры ограничений целостности: 

- Уникальность названия команды в `lfl_teams`.
- Проверка, что сумма побед, ничьих и поражений равна количеству сыгранных игр.
- Обязательность имени игрока и руководителя.

## Ключевые возможности

Проект демонстрирует навыки:

- Нормализация и проектирование даталогической модели.
- Использование внешних ключей и CHECK‑ограничений.
- Создание индексів для ускорения JOIN и типовых запросов.
- Реализация триггеров и процедур на PL/pgSQL.

Примеры реализованных сценариев:

- Автоматическое заполнение турнирных таблиц при добавлении команды.
- Автоматический пересчёт числа сыгранных игр на основе побед/ничьих/поражений.
- Проверка максимального количества игр в сезоне для каждой лиги.
- Генерация e‑mail руководителя по имени и фамилии.
- Проверка уникальности игрока в пределах одной команды.

## Индексы

Для оптимизации запросов созданы индексы, в том числе:

- По названию команды и уровню соревнований (`lfl_teams.team_name`, `competition_level`).
- По внешним ключам (`id_team` в `first_league`, `second_league`, `lfl_players`, `team_owners`).
- По участникам и дате матча (`match_results.team_1_id`, `team_2_id`, `match_date`).
- По фамилии и имени игрока для ускорения поиска однофамильцев.

## Триггеры

Основные триггеры:

- Автозаполнение `first_league` / `second_league` при вставке команды в `lfl_teams`.
- Автоматический расчёт `games_played` на основе полей `wins`, `draws`, `loses`.
- Контроль максимального числа игр в сезоне по формуле \((количество\_команд - 1) × 2\).
- Проверка уникальности игрока в рамках команды.
- Автоматическое заполнение `mail` для руководителей по шаблону `<имя>.<фамилия>@yandex.com`.

## Хранимые процедуры

Реализованы процедуры на PL/pgSQL:

- Обновление статистики команды (победы, ничьи, поражения, очки) по названию для первой и второй лиги.
- Добавление новой команды с учётом её лиги и инициализацией записи в соответствующей таблице лиги.
- Добавление игрока в команду по имени команды.
- Получение статистики команды (игры, победы, ничьи, поражения, очки) по `id_team`.

## Примеры запросов

Подготовлено 20 SQL‑запросов для анализа данных лиги, среди них:

- Вывод таблиц лиг по очкам.
- Список команд, претендующих на повышение или вылет.
- Поиск «дублевых» составов.
- Статистика игроков и руководителей конкретной команды (например, «Орион», «Дибиси»).
- Команда с максимальным количеством очков в обеих лигах.
- Команды без побед и команда с наибольшим числом пропущенных голов.

## Как запустить

1. Создайте базу данных в PostgreSQL.
2. Выполните скрипт с `CREATE TABLE` и ограничениями.
3. Выполните скрипт с триггерами, функциями и процедурами.
4. Выполните скрипт с `INSERT` и `CALL` для заполнения тестовыми данными.
5. Запустите набор запросов для проверки работы модели.
